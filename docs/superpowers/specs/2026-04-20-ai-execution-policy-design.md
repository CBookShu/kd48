# AI 执行规范设计文档

**日期**: 2026-04-20  
**主题**: AI Execution Policy  
**状态**: 设计中

---

## 目标 (Goal)

为 kd48 项目建立一套完整的 AI 执行规范，确保所有 AI 参与的工作都遵循一致的质量标准，同时通过明确的豁免机制保持效率。

核心原则：**无豁免即完整流程** — 所有任务默认遵循 superpowers 全流程，只有明确列入豁免清单的场景可以简化。

---

## 设计 (Design)

### 1. 标准工作流

```
┌─────────────────────────────────────────────────────────────────────┐
│                          任务开始                                    │
└─────────────────────────┬───────────────────────────────────────────┘
                          │
                          ▼
    ┌──────────────────────────────────────────┐
    │         是否有明确豁免？                   │
    └──────────┬─────────────────────┬─────────┘
               │ 否                  │ 是
               ▼                     ▼
    ┌────────────────────┐   ┌────────────────────┐
    │   完整流程          │   │   简化流程         │
    │                    │   │                    │
    │ 1. brainstorming   │   │ • 简要确认目标      │
    │    (设计文档)       │   │ • 直接执行          │
    │        ↓           │   │ • 验证通过即可      │
    │ 2. using-git-      │   │                    │
    │    worktrees       │   │ 适用于豁免清单场景   │
    │    (隔离环境)       │   └──────────┬─────────┘
    │        ↓           │              │
    │ 3. writing-plans   │              │
    │    (实现计划)       │              │
    │        ↓           │              │
    │ 4. TDD 开发        │              │
    │        ↓           │              │
    │ 5. verification    │              │
    │        ↓           │              │
    │ 6. code-review     │◄─────────────┘
    └────────┬───────────┘
             ▼
    ┌────────────────────┐
    │      完成           │
    └────────────────────┘
```

### 2. 豁免清单（明确的简化场景）

以下场景可跳过 `brainstorming` 和 `writing-plans`，直接进入执行和验证：

| 序号 | 场景 | 豁免理由 | 仍需执行 |
|------|------|---------|---------|
| 1 | **纯注释修改** | 不影响代码逻辑，仅增强可读性 | 验证文档生成正确 |
| 2 | **配置文件更新** | 无代码逻辑变更，仅需验证格式 | 验证配置可加载、服务可启动 |
| 3 | **测试补充** | 不改变实现，仅增加覆盖率 | 遵循 TDD、测试必须通过 |
| 4 | **格式化/lint 修复** | 机械性变更，工具自动完成 | 工具执行、验证通过 |
| 5 | **依赖升级** | 版本号变更，无代码修改 | 构建通过、测试通过、检查 breaking changes |
| 6 | **简单重构** | IDE 自动重构，行为保证不变 | 验证行为未变（测试通过）|
| 7 | **原型验证** | 明确标记为不合并主分支 | 仅需 brainstorming 简要确认，豁免 review |

> **豁免使用方式**: 任务开始时由 AI 主动声明豁免理由，用户确认后即可进入简化流程。

### 3. AI 行为规范

#### 3.1 编码标准

- **风格一致性**: 新代码必须与项目现有代码风格保持一致
- **文档义务**: 每个 exported 符号必须有 GoDoc 注释
- **错误处理**: 不得忽略 error，必须显式处理或传播
- **并发安全**: 并发代码必须考虑竞态条件，敏感区域需加注释说明

#### 3.2 测试要求

- **TDD 强制**: 新功能和 bug 修复必须先写测试
- **覆盖率**: 不追求 100%，但核心业务路径必须有测试覆盖
- **测试质量**: 测试应验证行为而非实现（避免过度 mock）

#### 3.3 文档要求

- **设计文档**: `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- **计划文档**: `docs/superpowers/plans/YYYY-MM-DD-<feature>-plan.md`
- **代码注释**: 复杂逻辑必须说明 **Why** 而非 **What**

### 4. 质量门禁检查点

#### 自动检查（每次提交前必须执行）

```bash
# 构建检查
go build ./...

# 测试检查
go test ./...

# 静态分析
go vet ./...
```

#### 手动确认（AI 必须主动报告）

- [ ] 变更范围已清晰说明
- [ ] 测试覆盖情况已评估
- [ ] review 检查清单已完成（非豁免场景）

### 5. 与 Superpowers 技能的整合

| 阶段 | 对应 Skill | 输出物 | 豁免? |
|------|-----------|--------|-------|
| 需求澄清 | `superpowers:brainstorming` | 设计文档 (specs/) | 豁免清单场景可简化 |
| 环境隔离 | `superpowers:using-git-worktrees` | Git 工作树 | 豁免清单场景可跳过 |
| 计划制定 | `superpowers:writing-plans` | 计划文档 (plans/) | 豁免清单场景可跳过 |
| 代码实现 | `superpowers:test-driven-development` | 代码 + 测试 | 永不豁免 |
| 完成验证 | `superpowers:verification-before-completion` | 验证报告 | 永不豁免 |
| 代码审查 | `superpowers:requesting-code-review` | 审查反馈 | 原型可豁免 |

### 6. 错误处理和回退机制

| 场景 | 处理方式 |
|------|---------|
| 用户拒绝豁免申请 | 立即转入完整流程 |
| 简化流程中测试失败 | 必要时退回完整流程分析原因 |
| 审查发现严重问题 | 返回 brainstorming 重新审视设计 |
| 依赖升级导致不兼容 | 创建完整计划处理 breaking changes |
