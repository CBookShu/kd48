## 1. 愿景、边界与文档关系

### 1.1 产品场景：大厅与游戏内

典型游戏产品划分为两个 **场景**（可与客户端界面/路由一致）：

- **大厅（一个「位置」）**  
  玩家在此 **查看活动**、**充值**、**查看房间信息**（含档位如初级/中级/高级房等）、**选择房间并进入**。大厅侧重 **导航与运营入口**，不承载局内权威状态。

- **游戏内（另一种场景）**  
  一般为 **对局发生处**：玩法、回合、胜负等 **以局内为焦点**。局内 **也可能展示部分活动信息**，但 **主体仍是游戏本身**；与大厅的职责边界需在接口与状态机上保持清晰。

### 1.2 玩法链路：开房间、匹配与对局

- **房间（逻辑）**：从大厅进入后形成 **归属关系**；在 **未开桌 / 未进入对局前** 处于 **匹配** 阶段；允许 **主动退房** 取消匹配并 **回到大厅**；异常断线后应在约定规则下 **回到原房间** 继续参与匹配（超时、心跳、多端等细则 **待定**）。
- **游戏服务 / 游戏内对局**：匹配成功后，玩家 **迁入对局**（进入 **游戏内** 场景）；未来支持 **回合制** 等玩法，对局侧为 **有状态服务** 演进方向。
- **未来**：更多 **活动形态**（如签到、任务、排行赛等仅为示例）、**简单客户端能力** —— 路线图占位，范围 **需产品确认**。

### 1.3 技术愿景（当前共识）

- **分布式**：整体可 **扩缩容**。
- **服务分类**：
  - **无状态服务**：主要与 **MySQL、Redis** 交互，水平扩展为主。
  - **有状态服务**：**房间** 明确为 **有状态服务（服务池）**；**游戏/对局** 为另一类有状态演进目标。有状态池的 **扩缩、路由、故障恢复** 在实现阶段设计。
- **治理与配置**：**玩家流量染色**、**AB**、**业务配置打表** 等 —— 横切能力，分阶段落地（见第 6 节）；其中 **染色的业务含义** 见 **§1.4**。

### 1.4 玩家流量染色（运营语义）

**染色**服务于 **精细化运营**：在配置 **一系列数据/策略**（活动参数、玩法数值、功能开关等）时，依据 **不同人群画像**（或实验分桶、标签体系）决定 **命中哪一套配置**，使同一功能对不同玩家呈现 **差异化、可度量** 的行为。技术上与网关 **流量染色**、**AB**、**打表下发** 协同实现；画像与标签模型 **实现阶段细化**。

### 1.5 文档体系与根目录 `spec.md`

- **当前**：根目录 **`spec.md`** 为 **M0 基座技术说明（草稿）**，实现细节的重要参考；本路线图 **不写重复细则**，必要时 **引用 `spec.md` 章节**。
- **演进方向**：在 **`docs/`** 下 **持续完善** 项目文档（如本路线图、`docs/superpowers/specs/` 等专题），形成 **结构化文档体系**；**长期目标** 是由该体系 **承接并逐步替代** 根目录 `spec.md` 的「单一总 spec」角色（迁移期可 **双写** 或在 `spec.md` 顶部保留 **索引指向 docs**，直至内容迁完再下线或缩为指针页）。
- **冲突处理**：在 **第 8 节** 或阶段表中维护 **「对 `spec.md` / 对 docs 主文档的修订建议」**，经 **负责人确认后** 再合并，避免静默分叉。
- **网关与后端（Ingress、Etcd 元数据、Watch）**：专题规格见 [`superpowers/specs/2026-04-13-gateway-backend-connection-design.md`](./superpowers/specs/2026-04-13-gateway-backend-connection-design.md)（**§11** 动态配置，**§11.12** 实现与可靠性）。

## 2. 业务域总览

顺序为 **推测优先级**，**需产品确认** 后可调整。

| 顺序 | 业务域 | 价值摘要 | 服务形态（路线图级） |
|------|--------|----------|----------------------|
| 0 | 账号与会话 | 注册登录、安全、顶号/多端策略 | 与现有 User + Redis Session 对齐；无状态 + Redis |
| 1 | 大厅 | 活动、充值、房间信息与档位、选房进房；运营与导航入口 | **无状态** + MySQL/Redis |
| 2 | 房间 | 归属、匹配、开桌前退房与重连语义 | **有状态服务池**（见 3.2） |
| 3 | 配置与实验 | 染色、AB、打表配置下发 | **横切**；与网关治理能力协同 |
| 4 | 游戏内 / 对局 | 匹配后局内玩法（含未来回合制）；可与大厅活动展示弱耦合 | **有状态服务**（演进） |
| 5 | 活动 | 签到、任务、排行赛等（示例）；大厅侧重入口，局内可少量透出 | 未来；依赖账号、配置与染色 |
| 6 | 轻量客户端 | 待定范围 | 占位，不与后台阶段强绑 |

## 3. 架构原则（路线图级）

### 3.1 无状态与有状态

- **无状态服务**：不依赖单机内存作为唯一真理；状态在外置 MySQL/Redis；实例可任意扩缩。
- **有状态服务**：**房间服务池**、未来 **游戏对局服务** —— 进程/节点承载权威状态；扩缩依赖 **分区、粘性路由、服务发现** 等（**实现阶段定**）。

### 3.2 房间服务（定调）

- **房间 = 有状态的服务器（服务池）**。
- **单进程单房** 与 **一进程多房** 等部署形态：**实现阶段再定**（路线图 **不锁定**）。
- **领域行为（已共识）**：进房即归属；开桌前可退房回大厅；断线后应能回房继续匹配（规则待定）。

### 3.3 匹配成功 → 进入对局（衔接）

采用 **丙：并列待定，实现阶段再选**：

- **同步路径**：匹配结果在一次调用链内创建对局并返回对局标识/参数。
- **异步路径**：匹配结果进入队列或事件，对局服务异步拉起，客户端通过推送/轮询获取。

正式选型时补充：一致性、延迟、失败重试与幂等等要求。

## 4. 阶段划分与交付边界

阶段编号可与产品对外版本另做映射；此处 **M0** 与 `spec.md` 中 M0 对齐。

| 阶段 | 产品叙事 | 技术交付（概要） | 对 `spec.md` 的关联 |
|------|----------|------------------|---------------------|
| **M0** | 统一入口、账号与会话、可观测、本地可跑 | 网关 WS、gRPC、User、Etcd、MySQL、Redis、OTel、sqlc/migrate | **当前基座草稿** |
| **M1** | 架子可扩、协作不断档 | compose/README、第二服务接入套路、最小 CI、配置与日志约定；WS 信封与 §4.3 可对齐 | §8 本地子集、§10 搭架子 |
| **M2** | 大厅可用（活动/充值/房间信息与选房进房） | 大厅 **无状态** 服务 + DB/缓存；染色/AB/打表 **最小可用**（支撑人群命中配置） | 建议修订：大厅域、配置横切、染色运营语义 |
| **M3** | 房间：归属、匹配、开桌前退房与重连（规则待定） | **房间有状态服务池**；路由与故障恢复 **实现阶段设计** | 建议修订：有状态房间、网关长连接关系 |
| **M4** | 游戏内对局：匹配后迁入游戏服务 | **有状态** 游戏/回合服务（可先简化）；衔接 **§3.3 丙** 选型落地；局内活动展示 **按需** | 建议修订：游戏内场景、对局域、状态与持久化 |
| **M5+** | 活动深化、轻量客户端 | 依赖前置阶段数据、染色与配置 | 需产品确认范围与顺序 |

## 5. 错误处理、测试与发布（路线图占位）

本节只约定 **各阶段应交付的「质量门槛」类型**，不替代具体测试计划。

### 5.1 错误处理

- **对内**：继续遵循 `spec.md` **§4.3**（gRPC `Status`、业务码、网关 JSON 收口）。
- **对外**：客户端可见错误与 **大厅/房间/对局** 状态机一致；有状态服务需定义 **可恢复错误** vs **致命错误**（实现阶段细化）。

### 5.2 测试

- **M0～M1**：单测 + 关键路径集成（网关 ↔ User ↔ Redis/MySQL）。
- **M2～M3**：大厅与房间 **集成与混沌场景**（断线、退房、重复进房）— **用例与数据** 实现阶段补全。
- **M4+**：对局 **状态机与回放/对账** 类测试（视玩法定）。

### 5.3 发布与日历

- 本路线图 **不写死日历**；产品确认里程碑后，可在本文件或项目管理工具中增加 **季度/版本** 列。

## 7. 风险与待定项（非穷尽）

- **有状态房间 / 对局** 的 **故障迁移、持久化、脑裂** —— 实现阶段必须单独立项。
- **断线重连**：TTL、心跳、多端是否共房 —— **待定**。
- **活动与付费** 若引入 —— 合规与风控 **需产品/法务输入**。
- **衔接 丙** 落地时须在 **同步 vs 异步** 间做明确选择与 SLO。
