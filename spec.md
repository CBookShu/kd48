## 1. 概述
### 1.1 核心架构理念
🟢 `[✅ 已完成]`
本系统采用**“基于显式契约与编译期安全的云原生 Go 微服务硬核基座”**架构范式。
系统坚决拒绝运行时魔法（不使用运行时 ORM、不使用运行时 DI 框架），全面拥抱“代码生成”体系（Protobuf、sqlc），在编译期拦截低级错误，保证系统在高并发下的行为绝对可控、可预测、易排障。控制权绝对掌握在开发者手中。

### 1.2 架构演进策略
* **当前阶段（M0）** 🟢 `[✅ 已完成]`：以同步 gRPC 链路为主干，采用**显式构造函数注入**完成依赖组装，打通全链路管控与追踪。
* **未来阶段（M1+）** 🟡 `[⚠️ 待规划]`：引入消息队列（MQ）作为异步车道，处理事件驱动、解耦与削峰。

## 2. 工程结构与基础设施
🟢 `[✅ 已完成]`
### 2.1 代码仓库管理
* **模式**：采用 `go work` (Monorepo) 单仓多模块管理。
* **规范**：网关、所有业务微服务、公共工具包均作为独立 Go Module 存在于同一代码仓库中，通过本地 `go.work` 互相引用，杜绝版本不同步问题。

### 2.2 基础中间件选型
* **注册中心**：**Etcd**（承载微服务节点注册与发现）。
* **配置中心**：**本地配置文件挂载**（基于 `viper` 读取 `/conf/config.yaml`）。前期不引入重度配置中心，通过 K8s ConfigMap 或本地卷实现环境隔离，保持架构极简。

### 2.3 多数据源与连接池路由（MySQL / Redis）
🟡 `[⚠️ 待规划/部分落地]`  
* **需求**：在 **执行查询或 Redis 命令之前**，按 **`routing_key`（抽象路由键，与物理 Redis key 解耦）** 选择 **命名连接池**；规则为 **最长前缀匹配（LPM）**。  
* **未命中**：**无候选规则时 `Resolve*` 须报错**（策略 **A**）；**禁止** 隐式默认池；全集兜底须在路由表中 **显式** 配置 **`prefix: ""`**（详见专项 §3）。  
* **单一信源**：[`docs/superpowers/specs/2026-04-17-datasource-routing-and-pools.md`](docs/superpowers/specs/2026-04-17-datasource-routing-and-pools.md)。

## 3. 流量网关设计
🟢 `[✅ 已完成]`
### 3.1 选型与定位
* **选型**：**Go 语言自研轻量级 API Gateway**（不采用 Kong/APISIX 等重型黑盒网关）。
* **定位**：作为系统唯一外部入口，负责协议转换与基础治理。

### 3.2 协议转换映射
* **外部短连接**：客户端 `HTTP/HTTPS` -> 网关转换 -> 内部 `gRPC 一元调用`。
* **外部长连接**：客户端 `WS/WSS` -> 网关接手连接 -> 内部转换为 `gRPC 双向流`。

### 3.3 核心网关治理能力
* **TLS 卸载**：在网关层统一终止 HTTPS/WSS。
* **统一鉴权（Redis 极简 Session 机制）**：
  * **摒弃 JWT**：鉴于系统核心链路为 WebSocket 长连接，JWT 存在无法主动吊销、黑名单维护成本高等反模式问题，故不采用。
  * **签发**：User Service 在 **登录成功**或**注册成功（注册即登录）**后，使用 `crypto/rand` 生成 32 字节安全随机字符串作为 Token，以 `user:session:{token}` 为 Key（与实现对齐）、`userId:username` 为 Value 存入 Redis 并设定 TTL；两种路径使用同一套会话模型，便于网关与强踢逻辑一致。
  * **校验**：网关层拦截请求，仅以该 Token 去 Redis 查询，命中且未过期即放行（返回 200 或完成 WS 握手），否则直接返回 401 拒绝穿透后端。
  * **强踢机制（核心优势）**：当需要踢人下线（如封号、异地登录）时，服务端直接 `DEL` 对应的 Redis Key。网关感知到失效后，主动断开该用户的 WebSocket 长连接，强制客户端走正常密码校验流程。
  * **重复登录 / 顶号与会话策略** 🟡 `[⚠️ 待规划]`：产品层需明确「新登录是否使旧会话失效、是否允许多终端同时在线」等规则；实现上与会话 Key 设计及网关断连协同，**具体方案待与业务迭代一并敲定**。
* **网关安全与连接治理（防爆破/防探测）** 🟢 `[✅ 已完成]`：网关作为第一道防线，对未认证连接采取“零容忍”断开策略。
  * **握手超时（未认证阶段）**：在 WebSocket 应用层循环中，**尚未完成认证前**，每次阻塞读取上行消息前设置**短读限时**（如 10s）；若超时仍未收到下一帧则断开连接。已认证后**取消**该短读限时，避免误伤长连接。可选后续再单独定义已认证连接的 idle/心跳策略。
  * **未认证期强断**：未登录状态下，若发生非文本帧、JSON 解析失败、越权调用业务接口、探测不存在路由等行为，视为恶意探测，发送错误提示后立即物理断开连接。
  * **认证失败强断**：Login/Register 接口返回业务错误（如密码错误），直接物理断开连接，杜绝网关层被暴力破解。
  * **注册即登录（连接语义）**：`Register` 与 `Login` 在网关 WebSocket 连接上**业务成功**后，连接均进入「已认证」状态，可继续调用其它业务方法；User Service 在 Register 成功时完成与 Login **同等**的 Redis 会话签发（见上「签发」）。
  * **宽容降级**：仅当用户“已认证”时，常规业务参数错误才保持连接不断开，允许前端重试。
* **长连接优化**：WS 握手成功建立 TCP 连接后，后续业务消息无需再携带 Token，依靠底层连接标识寻址。
* **长连接会话保持**：针对 WS/gRPC 流，基于用户标识计算**一致性哈希**，确保特定用户的连接永远路由到同一台有状态后端节点。
* **灰度发布路由**：支持流量染色，网关根据请求 Header 中的版本标识，将长短连接流量精准路由到特定的灰度节点池。

## 4. 服务通信与契约规范
🟢 `[✅ 已完成]`
### 4.1 接口契约管理 (IDL)
* **管理模式**：**集中式 Proto 模块**。在 Monorepo 根目录建立独立 Go Module（如 `api/proto`）。
* **依赖规范（当前落地）**：业务微服务必须且只能引用该集中模块，保证接口定义的单一信源（SSOT）。
* **网关与 IDL 的演进方向** 🟡 `[⚠️ 待规划]`：目标是避免「业务侧频繁调整 proto / 生成代码」时网关模块被迫同步修改与重编译；具体技术路径（如运行时描述符、反射调用、独立 BFF 契约层等）待选型，**不与当前 M0 已跑通实现强行绑定**。

### 4.2 服务发现与负载均衡
* **桥接组件**：引入成熟的第三方 `etcd resolver` 库，不造轮子。
* **调用规范**：所有微服务发起 gRPC 调用时，统一使用 `etcd:///服务名` 格式作为 Target，由 Resolver 在后台 Watch Etcd 动态更新 gRPC 连接池。

### 4.3 跨协议错误透传规范
* **机制**：基于 gRPC 标准的 `Status` 结合自定义 `Error Details` 扩展。
* **内部服务**：抛出包含具体业务错误码（如 `10001`）和信息的 gRPC Status。
* **网关拦截**：网关捕获 gRPC 错误，解析 Details，组装为标准 HTTP JSON 结构 `{"code": 10001, "msg": "..."}`，并映射为对应的 HTTP 状态码返回前端。

## 5. 微服务内部架构规范
🟢 `[✅ 已完成]`
### 5.1 依赖注入与生命周期
* **当前方案（M0 显式组装）**：使用纯 Go 构造函数（如 `NewUserService(db *sql.DB, rdb *redis.Client)`）在 `main.go` 中**显式组装**依赖关系树。此方式零黑盒、启动报错堆栈绝对清晰。
* **演进预案**：当单个服务内部依赖超过 15 个以上导致 `main.go` 臃肿时，允许局部引入 **Google Wire**（编译期代码生成）优化，但严禁使用 Uber Fx 等运行时反射 DI。

### 5.2 业务微服务划分
* **已落地** 🟢 `[✅ 已完成]`：`User Service`（负责用户注册、密码校验、Redis Session Token 签发）。
* **待规划** 🟡 `[⚠️ 待规划]`：后续具体业务领域拆分（如：房间服务、礼物服务等）需根据业务迭代逐步落地。

## 6. 数据层设计
🟢 `[✅ 已完成]`
### 6.1 数据访问范式 (DAL)
* **方案**：**原生 SQL + `sqlc` 代码生成器**。
* **规范**：坚决不使用 GORM/Ent 等重型 ORM。开发人员编写 `.sql` 文件，由 sqlc 编译生成类型安全的 Go 代码，保证 SQL 性能透明可控。
* **数据库迁移**：采用 **`golang-migrate`** 管理 DDL 变更（Up/Down 成对），跟随业务代码一并提交。

### 6.2 存储与缓存选型
* **关系型数据库**：**MySQL 8.0**（要求开启 `parseTime` 以兼容 sqlc 的 `time.Time` 映射）。**连接池防断裂治理** 🟢 `[✅ 已完成]`：显式配置 `MaxOpenConns` 与 `MaxIdleConns`，且**必须设置 `ConnMaxLifetime`（建议 5 分钟）**，其值严格小于 MySQL 的 `wait_timeout`，防止闲置连接被 MySQL 侧掐断而 Go 侧误用导致突发崩溃。
* **多数据源与按逻辑 Key 路由** 🟡 `[⚠️ 待规划]`：演进上需支持**多个** Redis / MySQL 实例（或库），由配置中的**逻辑名（Key）**解析到对应连接池，供不同领域或环境隔离使用；配置结构与公共封装（如 `pkg` 内 registry）待设计。
* **缓存与会话中间件**：**Redis**。
  * **定位**：不仅是缓存层，更是核心的**有状态会话管理中心**。
  * **客户端**：统一采用 `github.com/redis/go-redis/v9`。
  * **规范**：所有涉及状态判断、强踢机制的数据必须以 Redis K-V 为准，不依赖 DB 实时查询。

## 7. 可观测性体系
🟢 `[✅ 已完成]`
### 7.1 分布式追踪
* **标准**：全面采用 **OpenTelemetry (OTel)**。
* **覆盖颗粒度**：
  * **网关层**：拦截入口，生成 Root Span。严格遵守 defer LIFO 机制，确保资源清理与日志打印在 `span.End()` 之前完成，保证断链日志完美挂载 TraceID。
  * **gRPC 层**：通过 OTel gRPC Interceptor 自动透传 TraceID 至 Metadata，并记录 RPC 耗时。
  * **数据层**：通过 OTel 的 database/sql Hook，自动记录每条 sqlc 生成 SQL 的执行耗时 Span。
* **存储与展示**：通过 OTLP 协议推送到 **Jaeger 或 Tempo**，实现完整的调用链瀑布图。

### 7.2 日志规范
* **核心组件**：采用 Go 1.21+ 原生 `log/slog` 作为门面接口。
* **底层实现**：封装 `slog.Handler` 接入 **Zap** 高性能底层（`pkg/logzap`），兼顾原生 API 的规范性与 Zap 的吞吐量。
* **双轨制输出与格式规范** 🟢 `[✅ 已完成]`：采用 `zapcore.Tee` 机制实现多目标分发。终端输出采用 `ConsoleEncoder`（平铺+高亮颜色，讨好人类肉眼排障）；文件输出采用 `JSONEncoder`（结构化，讨好 ELK/Loki 机器）。强制注入 OTel `trace_id`，时间格式统一为 ISO8601，调用栈采用 `ShortCaller`（去除绝对路径噪音）。
* **文件隔离与滚动切割** 🟢 `[✅ 已完成]`：引入 `lumberjack` 实现按大小自动切割与旧日志压缩；基于 `ServerConf.Name` 动态拼接日志文件绝对路径，从物理层面隔离不同微服务（如网关与用户服务）的日志流，杜绝多进程并发写同一文件导致的日志穿插错乱。

## 10. M0「搭架子」优先级建议（尽快可跑、可扩）
🟡 `[⚠️ 待规划]`

**目标**：新人一条命令（或一份最短文档）能起依赖、跑通网关 + 至少一个业务服务，并看清「加第二个服务」的重复模式。

**建议优先顺序（由高到低）**：

1. **本地依赖与启动闭环**：`docker compose`（或等价方式）一键拉起 MySQL、Redis、Etcd；README 写清配置路径、migrate、注册与访问顺序。对应补齐 **§8** 中的「本地开发」最小子集即可，不必先上完整 K8s。
2. **多服务扩展样板**：在 Spec 与代码中固定「第二个 gRPC 服务」的接入套路（etcd 服务名、网关路由注册、WS `method` 约定），哪怕第二个服务仍是空实现。避免长期只有 User 一条路径导致后续返工。
3. **工程化骨架**：根级 `Makefile` 或脚本覆盖常用动作（`sqlc generate`、`migrate`、`buf`/`protoc` 若已采用则统一入口）、**最小 CI**（`go test` / `go build` 全 module）。保证架子提交后协作不断档。
4. **契约与对外消息形态**：WS 信封、错误码与 **§4.3** 的对齐方式（可仍为 JSON + 现有结构），避免前端与各服务各自约定；具体是否统一外层 proto **可后于** 1～3 步。
5. **§9 演进项**：重复登录策略、网关动态 IDL、多数据源 registry 等，在「能稳定跑、能加服务」之后再逐项立项实现，避免阻塞架子交付。
